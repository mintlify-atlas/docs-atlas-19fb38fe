## Overview

NoOCR's visual search feature allows you to find relevant document pages using natural language queries—without requiring OCR or text extraction. This is powered by **ColPali**, a vision-language model that creates multi-vector embeddings directly from document images.

### How It Works

1. **Query Processing**: Your text query is encoded into a multi-vector embedding by ColPali
2. **Vector Search**: The query embedding is compared against document page embeddings in LanceDB
3. **Result Ranking**: Pages are ranked by cosine similarity and the top matches are returned
4. **Image Retrieval**: Matching pages are retrieved from the HuggingFace dataset as base64-encoded images

<Note>
  Visual search works on the actual appearance of documents, not just text. This means it can find information in tables, charts, handwritten notes, and complex layouts that traditional OCR might miss.
</Note>

## Search Implementation

The search functionality is implemented in the `SearchClient` class (`np_ocr/search.py:67-139`):

```python
class SearchClient:
    def __init__(self, storage_dir: str, vector_size: int, base_url: str, token: str):
        self.storage_dir = storage_dir
        self.vector_size = vector_size
        self.colpali_client = ColPaliClient(base_url, token)

    def search_images_by_text(self, query_text, case_name: str, user_id: str, top_k: int):
        # Connect to LanceDB
        lance_client = lancedb.connect(f"{self.storage_dir}/{user_id}/{case_name}")
        tbl = lance_client.open_table(case_name)

        # Get query embedding from ColPali
        query_embedding = self.colpali_client.query_text(query_text)
        multivector_query = np.array(query_embedding["embedding"])
        
        # Search for similar vectors
        search_result = tbl.search(multivector_query)\
            .limit(top_k)\
            .select(["index", "pdf_name", "pdf_page"])\
            .to_list()

        return search_result
```

## ColPali Integration

ColPali is a vision-language model specifically designed for document retrieval. NoOCR uses a separate ColPali service for encoding queries and document images.

### ColPali Client

The `ColPaliClient` class handles communication with the ColPali service:

```python
class ColPaliClient:
    def __init__(self, base_url: str, token: str):
        self.base_url = base_url
        self.headers = {"Authorization": f"Bearer {token}"}

    def query_text(self, query_text: str):
        response = requests.post(
            f"{self.base_url}/query",
            headers=self.headers,
            params={"query_text": query_text}
        )
        response.raise_for_status()
        return response.json()

    def process_pil_image(self, pil_image):
        buffered = io.BytesIO()
        pil_image.save(buffered, format="JPEG")
        files = {"image": buffered.getvalue()}
        response = requests.post(
            f"{self.base_url}/process_image",
            files=files,
            headers=self.headers
        )
        response.raise_for_status()
        return response.json()
```

### Multi-Vector Embeddings

Unlike traditional single-vector embeddings, ColPali generates **multi-vector embeddings** where each embedding is a list of vectors. This captures more nuanced information about document structure and layout.

<Accordion title="What are multi-vector embeddings?">
  Traditional embedding models generate a single vector per document (e.g., 768-dimensional vector). ColPali generates multiple vectors per document page, where each vector captures different aspects of the page content.

  For example, a document page might have:
  - Vectors for header text
  - Vectors for body paragraphs
  - Vectors for table data
  - Vectors for images or charts

  During search, the query is also encoded as multiple vectors, and similarity is computed across all vector pairs. This enables more precise matching of specific document regions.
</Accordion>

## LanceDB Vector Storage

NoOCR uses **LanceDB**, a disk-based vector database optimized for multi-modal AI applications.

### Why LanceDB?

- **Efficient multi-vector search**: Native support for lists of vectors
- **Disk-based storage**: No need to load entire indexes into memory
- **Columnar format**: Fast filtering and projection of results
- **Cosine similarity**: Built-in support for the similarity metric ColPali expects

### Index Structure

During case ingestion, document pages are indexed with this schema:

```python
schema = pa.schema([
    pa.field("index", pa.int64()),           # Global page index
    pa.field("pdf_name", pa.string()),        # Source PDF filename
    pa.field("pdf_page", pa.int64()),         # Page number in PDF
    pa.field("vector", pa.list_(             # Multi-vector embedding
        pa.list_(pa.float32(), vector_size)
    )),
])

lance_client = lancedb.connect(f"{storage_dir}/{user_id}/{case_name}")
tbl = lance_client.create_table(case_name, schema=schema)
```

### Creating the Index

After all pages are added, a cosine similarity index is created:

```python
tbl.create_index(metric="cosine")
```

This enables fast approximate nearest neighbor (ANN) search for queries.

## Search Workflow in Detail

Here's what happens when you call `POST /search` (from `np_ocr/api.py:169-238`):

```python
@app.post("/search", response_model=SearchResponse)
def ai_search(user_query: str = Form(...), user_id: str = Form(...), case_name: str = Form(...)):
    # 1. Validate inputs
    validate_identifier(user_id, "user_id")
    validate_identifier(case_name, "case_name")

    # 2. Check case exists
    case_info_path = os.path.join(settings.STORAGE_DIR, user_id, case_name, settings.CASE_INFO_FILENAME)
    if not os.path.exists(case_info_path):
        raise HTTPException(status_code=404, detail="Case info not found.")

    # 3. Search vector index
    search_results = search_client.search_images_by_text(
        user_query,
        case_name=case_name,
        user_id=user_id,
        top_k=settings.SEARCH_TOP_K,
    )

    # 4. Load dataset
    dataset_path = os.path.join(settings.STORAGE_DIR, user_id, case_name, settings.HF_DATASET_DIRNAME)
    dataset = load_from_disk(dataset_path)

    # 5. Retrieve images and convert to base64
    search_results_data = []
    for point in search_results:
        score = point["_distance"]
        index = point["index"]
        image_data = dataset[index]["image"]
        pdf_name = dataset[index]["pdf_name"]
        pdf_page = dataset[index]["pdf_page"]

        buffered = BytesIO()
        image_data.save(buffered, format="JPEG")
        img_b64_str = base64.b64encode(buffered.getvalue()).decode("utf-8")

        search_results_data.append(SearchResult(
            score=score,
            pdf_name=pdf_name,
            pdf_page=pdf_page,
            image_base64=img_b64_str
        ))

    return SearchResponse(search_results=search_results_data)
```

## Search Result Scores

Search results include a `score` field representing the **cosine distance** between the query and document embeddings:

- **Lower scores** = more similar (better match)
- **Score range**: typically 0.0 to 2.0
- **Threshold**: scores below 1.0 usually indicate good matches

<Warning>
  LanceDB returns `_distance` (cosine distance), not cosine similarity. To convert to similarity: `similarity = 1 - (distance / 2)`.
</Warning>

## Performance Considerations

### Indexing Performance

From the ingestion implementation (`np_ocr/search.py:73-123`):

```python
def ingest(self, case_name: str, dataset, user_id: str, batch_size: int = 50):
    # Process images in batches of 50
    batch = []
    for i in range(len(dataset)):
        image = dataset[i]["image"]
        response = self.colpali_client.process_pil_image(image)
        image_embedding = response["embedding"]

        batch.append({
            "index": dataset[i]["index"],
            "pdf_name": dataset[i]["pdf_name"],
            "pdf_page": dataset[i]["pdf_page"],
            "vector": image_embedding,
        })

        if len(batch) >= batch_size:
            tbl.add(batch)
            batch = []
```

**Optimization tips:**
- Batch size of 50 balances memory usage and throughput
- Embedding generation is the bottleneck (network call to ColPali service)
- Consider deploying ColPali locally for faster ingestion

### Search Performance

Search queries are fast (typically < 200ms) because:
- LanceDB uses ANN indexes for approximate search
- Only top K results need full distance computation
- Cosine index is pre-computed during ingestion

## Error Handling

The search endpoint returns HTTP errors for common issues:

| Status Code | Error | Cause |
|-------------|-------|-------|
| 400 | Invalid user_id provided | `user_id` contains invalid characters |
| 400 | Invalid case_name provided | `case_name` contains invalid characters |
| 404 | Case info not found | Case doesn't exist or hasn't finished processing |
| 404 | Dataset for this case not found | Case dataset was deleted or corrupted |
| 500 | Failed to load case dataset | Disk I/O error or dataset corruption |
